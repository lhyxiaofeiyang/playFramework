      任何的互联网用户都可以在博客中发表评论，所以我们应该采取一些措施避免自动发布器等恶意软件所带来的大量的垃圾信息。为应用增加图形验证码是非常简单有效的保护表单的方法。


12.5.1 生成验证码#

      Play是全栈式的Web框架，内置了Web应用的大部分典型需求。使 用play.lib.Images工具类可以非常容易地生成图形验证码，并将其写入到HTTP响应中。在Play中只需创建一个Action，就能轻松实 现图形验证码功能。


补充：

图形验证码Action与常规Action的区别在于返回的结果是二进制流，并不是HTML响应。


      编辑Application控制器，为其增加captcha Action： 

public static void captcha() {
    Images.Captcha captcha = Images.captcha();
    renderBinary(captcha);
}

      因为Images.Captcha类实现了java.io.InputStream，所以能够直接将captcha对象传递给renderBinary()方法。


提示：

不要忘记了导入play.libs包。


      最后在/yabe/conf/routes文件中为验证码Action配置路由规则： 

GET     /captcha                                Application.captcha

      就完成了生成验证码的操作。在浏览器中打开http://localhost:9000/captcha查看生成的验证码的效果，每次刷新都会生成不同的随机验证码，如图12.13所示：

12.5.2 管理验证码状态#

      为了确保用户输入的验证码的正确性，我们必须事先将写入到图形验证码中的随机文本进行保存，然后在表单提交的时候进行匹配检查。读者可能会想到能否在生成验证码的时候，将文本信息储存到用户的session中，方便读取。这样做虽然可行，但有两个比较致命的缺点：

  1. Play中的session是以cookie的形式储存的。这种设计出于架构方面的考虑，有自身的优势但也存在很多潜在的问题。写到cookie中的数据已经被签名（所以用户不能改写），但并没有加密。如果将验证码的文本信息写到cookie中，那么任何人都可以轻易地读取用户cookie进行分析，存在安全隐患。
  2. 由于Play是一个无状态的框架，是在完全无状态的环境下管理事务。如果一个用户同时打开两个包含验证码的博客页面，系统程序就不得不对每一个表单的验证码进行跟踪。

      基于以上几点的考虑，Play的解决方案是在服务器端存储验证码的key。由于验证码的key是临时数据，可以使用Play的缓存进行存储。为了便于之后获取验证码信息，程序需生成唯一的id，此id会以隐藏域的方式添加到每个表单，并参照已经生成的验证码（在有效期范围内）。


提示：

缓存的数据都具有有限的生命期，该特性也提升了验证码的安全性（假设验证码在10分钟内有效）。


      更改captcha Action，将验证码的key存入缓存: 

public static void captcha(String id) {
    Images.Captcha captcha = Images.captcha();
    String code = captcha.getText("#E4EAFD");
    Cache.set(id, code, "10mn");
    renderBinary(captcha);
}

      getText()方法不仅可以获取验证码的文本内容，还可以用来设置文本的颜色。


提示：

不要忘记导入play.cache包；


12.5.3 为表单添加验证码#

      在页面呈现博客文章和评论之前，就生成了验证码的唯一id。修改HTML表单，生成图片验证码，并将该id作为验证码的key存储在服务器的缓存中，同时将此id添加到表单的隐藏域中。

      修改Application.show Action： 

public static void show(Long id) {
    Post post = Post.findById(id);
    String randomID = Codec.UUID();
    render(post, randomID);
}

      修改/yabe/app/views/Application/show.html模板中的表单： 

...
<p>
    <label for="content">Your message: </label>
    <textarea name="content" id="content">${params.content}</textarea>
</p>
<p>
    <label for="code">Please type the code below: </label>
    <img src="@{Application.captcha(randomID)}" />
    <br />
    <input type="text" name="code" id="code" size="18" value="" />
    <input type="hidden" name="randomID" value="${randomID}" />
</p>
<p>
    <input type="submit" value="Submit your comment" />
</p>
...

12.5.4 检查验证码的有效性#

      验证码的唯一id(randomID)被添加至表单的隐藏域中，会随着请求一并提交。我们可以在postComment Action中将其获取，从而获取缓存中实际的验证码，最终和表单提交的验证码进行比较。更改postComment Action： 

public static void postComment(
        Long postId, 
        @Required(message="Author is required") String author, 
        @Required(message="A message is required") String content, 
        @Required(message="Please type the code") String code, 
        String randomID) 
{
    Post post = Post.findById(postId);
    validation.equals(
        code, Cache.get(randomID)
    ).message("Invalid code. Please type it again");
    if(validation.hasErrors()) {
        render("Application/show.html", post, randomID);
    }
    post.addComment(author, content);
    flash.success("Thanks for posting %s", author);
    Cache.delete(randomID);
    show(postId);
}

      以上操作就完成了验证码的校验工作。由于现在的postComment()方法具有多条数据验证，相应的就会有更多的错误信息。修改show.html模板，换一种方式来提示错误信息（我们只需要显示错误集中的第一条错误信息就足够了）： 

...
#{ifErrors}
    <p class="error">
        ${errors[0]}
    </p>
#{/ifErrors}
...


补充：

如果表单比较复杂，可以将错误信息写入yebe/conf/messages文件，并将错误在相应的字段旁打印。
